# Lead Intelligence

A local-only desktop app that turns your 30k-row Apollo.io export into a
searchable, filterable, scored lead database — no cloud, no monthly hosting,
your data never leaves your machine.

**Start here, then read in this order:**
1. `docs/ARCHITECTURE.md` — the stack, the data model, why local-only
2. `docs/DATA_PROFILE.md` — what's actually in your CSV and why each design
   decision was made
3. `docs/AGENT_BRIEF.md` — hand this to Claude Opus in Antigravity to
   continue the build

## Status at a glance

## Quickstart (Phase 1, works today)

```bash
cd backend/app/services
python3 -m pip install pandas
python3 data_cleaning.py --input ../../../database/imports/your_apollo_export.csv \
                          --output ../../../database/exports/clean_leads.csv
```

This dedupes, normalizes titles/companies, validates emails, and buckets
company size. See the report it prints for exact counts on your file.

## Build Log

This section is the running history of the project. Every deliverable gets
an entry: what was built, why, and how it connects to the rest of the app.

### 2026-07-22 — Project scaffold + Data Profile
**What:** Created the folder structure (`backend/`, `frontend/`, `database/`,
`docs/`, `config/`) and `docs/DATA_PROFILE.md`, generated by actually loading
`Apollo_scrapped_IT_leads.csv` (29,445 rows) with pandas and measuring fill
rates, duplicate counts, title variants, company-size distribution, and
geography.
**Why:** Every architecture decision below is grounded in what's really in
the file (e.g. `Email Confidence` is 0.03% filled → can't be trusted;
`# Employees` skews heavily toward 1-10 people → this is a startup-heavy
list, not an enterprise one) rather than assumptions carried over from a
generic CRM template.
**Connects to:** feeds directly into the field choices in
`ARCHITECTURE.md`'s data model and the rules in `data_cleaning.py` below.

### 2026-07-22 — Architecture decision consolidated
**What:** `docs/ARCHITECTURE.md` — merged the two prior proposals (cloud CRM
sketch vs. local/offline sketch) into one locked decision: fully local app
(React + FastAPI + SQLite), Tauri packaging, rule-based classification
before any LLM call.
**Why:** the data is personal contact information (names, emails, phones)
at meaningful volume — local-first removes an entire risk category and a
recurring cost for a single-user tool, with no loss of capability at this
scale (SQLite handles far more than 30k rows).
**Connects to:** this is the contract every later phase must follow —
Phase 1b's models, Phase 2's engine, and Phase 3's frontend are all designed
against the data model defined here.

### 2026-07-22 — Phase 1 data cleaning engine (built + verified)
**What:** `backend/app/services/data_cleaning.py`. Run standalone against
the real 29,445-row export. Result: **21,990 clean, unique leads** after
dropping 2,574 rows with no usable identity, 4,718 duplicate emails, and 163
duplicate (company, name) pairs. Adds `role_canonical`, `is_founder`,
`email_valid`, `company_match_key`, `company_size_bucket` columns; leaves
`industry_category`, `opportunity_type`, `priority_score` as explicit `None`
placeholders for Phase 2.
**Why:** the raw title column has 11+ spellings of "CEO" alone (see
`DATA_PROFILE.md`); company names contain invisible unicode characters that
silently break exact-match dedup; Apollo's own email-confidence score is
unusable (0.03% filled) so email validity had to be computed independently.
**Connects to:** output (`database/exports/clean_leads_sample.csv`) is the
direct input for the Phase 1b SQLite loader described in
`docs/AGENT_BRIEF.md`, and the placeholder columns are exactly what
Phase 2's classification engine will fill in — nothing about this file's
shape needs to change later, only those three null columns need values.

### 2026-07-22 — Phase 1b: SQLAlchemy models + database layer
**What:** `backend/app/database.py` (engine/session/Base), `backend/app/models/models.py`
(Company + Contact ORM models). Every field traced to actual CSV column fill rates.
Added `keywords`, `seo_description`, `seniority`, `departments` which were well-filled
(79-89%) in the real data.
**Why:** ARCHITECTURE.md data model needed to become code. Models include indexes on
all filterable columns (role, country, industry, size_bucket, opportunity_type, priority_score).
**Connects to:** consumed by the DB loader (below) and the FastAPI endpoints (below).

### 2026-07-22 — Phase 1b: DB loader with acceptance checks
**What:** `backend/app/services/load_to_db.py`. Reads `clean_leads_sample.csv`,
creates 17,226 Company rows (one per unique `company_match_key`), 21,990 Contact
rows linked by FK. Runs raw `sqlite3` acceptance checks after load.
**Why:** the clean CSV is flat — the loader splits it into the normalized two-table
model so company facts aren't duplicated per-contact.
**Connects to:** reads output of `data_cleaning.py`, writes `database/leads.db`
consumed by FastAPI.

### 2026-07-22 — Phase 1b: FastAPI skeleton with filtering
**What:** `backend/app/main.py` — endpoints: `GET /companies` (industry/country/size
filters + search), `GET /contacts` (role/founder/stage/opportunity/priority/seniority/
country/industry/size + search), `GET /stats`, `GET /filters`, `GET /contacts/{id}`,
`POST /classify`. All numbers verified against DATA_PROFILE.md.
**Why:** the React frontend needs a real API to consume, and filtering is core UX.
**Connects to:** reads `database/leads.db`, serves data to the React frontend.

### 2026-07-22 — Phase 2: YAML-driven Opportunity Engine
**What:** `config/opportunity_rules.yaml` (15 opportunity rules + 5 priority boosts),
`backend/app/services/classification_engine.py`. Classified all 21,990 contacts into
12 opportunity types (AI Automation 36.9%, SaaS Dev 16.4%, Cybersecurity 10.2%, etc.)
with priority scores 30-75 (avg 60, 4,708 high-priority).
**Why:** ARCHITECTURE.md mandates rule-based before LLM. All logic in editable YAML —
change rules without touching Python.
**Connects to:** reads companies+contacts from `leads.db`, writes `opportunity_type`
and `priority_score` back. Exposed via `POST /classify` endpoint.

### 2026-07-22 — Phase 3: React dashboard + Lead Finder
**What:** `frontend/` — Vite + React app with 4 pages: Dashboard (stat cards + bar
charts), Find Leads (wizard-style filter widget), Contacts (filterable table),
Companies (filterable table). Dark-mode glassmorphism design with Inter font.
**Why:** the API needs a UI. Lead Finder is the primary workflow: pick role +
opportunity + country, see matching leads or "Not Available".
**Connects to:** consumes all FastAPI endpoints. Runs on `localhost:5173`.
